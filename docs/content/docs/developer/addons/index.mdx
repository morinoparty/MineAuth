MineAuth provides a powerful API for external plugins to register custom HTTP endpoints. This guide will help you get started with addon development.

## Adding MineAuth API as a Dependency

### Gradle (Kotlin DSL)

Add the MineAuth API to your `build.gradle.kts`:

```kotlin
repositories {
    maven("https://repo.morinoparty.net/releases")
}

dependencies {
    compileOnly("party.morino.mineauth:api:1.0.0")
}
```

### plugin.yml Configuration

Add MineAuth as a dependency in your `plugin.yml`:

```yaml
name: MyPlugin
version: 1.0.0
main: com.example.myplugin.MyPlugin
depend:
  - MineAuth
```

## Quick Start

Obtain the `MineAuthApi` instance via `MineAuthApi.get(server)` and register your handlers in a single call:

```kotlin
class MyPlugin : JavaPlugin() {
    override fun onEnable() {
        // Look up MineAuthApi through the ServicesManager (null if MineAuth is absent)
        val api = MineAuthApi.get(server)
            ?: run { logger.severe("MineAuth not found"); server.pluginManager.disablePlugin(this); return }
        try {
            // Register all handlers under an explicit URL namespace
            val registration = api.register(this, "myplugin", MyHandler())
            logger.info("Mounted ${registration.endpoints.size} endpoints under ${registration.basePath}")
        } catch (e: EndpointRegistrationException) {
            logger.severe(e.message) // contains the full list of validation errors
            server.pluginManager.disablePlugin(this)
        }
    }
}

class MyHandler {
    @Get("/hello")
    @Public
    suspend fun hello(): Map<String, String> = mapOf("message" to "Hello, World!")
}
```

Key points:

- **All-or-nothing registration**: `register()` validates every endpoint up front. If any endpoint is invalid, nothing is mounted and an `EndpointRegistrationException` is thrown describing every error.
- **Explicit namespace**: the second argument (`"myplugin"`) becomes the URL namespace. It must match `[a-z0-9-]{2,32}` and must not be in use by another plugin. A two-argument overload derives the namespace from `plugin.name.lowercase()`, but an explicit namespace is recommended for clean URLs.
- **Explicit access declaration**: every endpoint method must be annotated with exactly one of `@Public` or `@Authenticated`. Registration fails otherwise, preventing endpoints from being exposed unintentionally.
- **Automatic cleanup**: registrations are unregistered automatically when your plugin is disabled. No manual cleanup is required.

## Basic Handler Example

Create a handler class with annotated methods:

```kotlin
class MyHandler {

    @Get("/hello")
    @Public
    suspend fun hello(): Map<String, String> {
        return mapOf("message" to "Hello, World!")
    }

    @Get("/users/{uuid}")
    @Public
    suspend fun getUser(@Path("uuid") uuid: UUID): UserData? {
        // Return user data by UUID
        return userService.findByUuid(uuid)
    }

    @Post("/items")
    @Authenticated(permission = "myplugin.items.create")
    suspend fun createItem(@Body item: ItemData): ItemData {
        // Create and return item
        return itemService.create(item)
    }
}
```

### Generated Endpoints

With the namespace `"myplugin"`, the endpoints will be:

| Method | Path |
|--------|------|
| GET | `/api/v1/plugins/myplugin/hello` |
| GET | `/api/v1/plugins/myplugin/users/{uuid}` |
| POST | `/api/v1/plugins/myplugin/items` |

## Project Structure

Recommended addon project structure:

```
my-addon/
├── src/main/kotlin/com/example/myaddon/
│   ├── MyAddon.kt           # Main plugin class
│   ├── handlers/
│   │   ├── ItemHandler.kt   # HTTP handlers
│   │   └── UserHandler.kt
│   ├── data/
│   │   ├── ItemData.kt      # Data classes
│   │   └── UserData.kt
│   └── services/
│       └── ItemService.kt   # Business logic
├── build.gradle.kts
└── src/main/resources/
    └── plugin.yml
```
